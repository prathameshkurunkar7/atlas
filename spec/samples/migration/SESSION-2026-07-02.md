### Key architecture decisions (differ from the older sample in this dir)

1. **Transport = plain TCP** (deferred secure comms). Source `qemu-nbd` binds its
   public IPv4; target `nbd-client` dials `source:port` directly. No SSH tunnel.
   Unencrypted — deliberate get-it-working-first shortcut.
2. **Cutover reuses `provision-vm`**, not a bespoke launcher. After hydration, the
   dm-clone collapses to the plain `atlas-vm-<uuid>` LV; the controller then runs
   `provision-vm` against that existing disk (its `snapshot_into` no-ops on an
   existing LV, so it reuses the hydrated bytes), injecting the new identity with
   `preserve_host_keys=1` and launching the jail+unit. ~200 lines of proven code
   reused instead of duplicated.
3. **`InjectingIdentity` is now controller-only** — just `allocate_ipv6` + record
   the new `/128`. The actual inject happens at cutover via provision-vm.

### Bugs found + fixed (all in the new code)

1. `pool.usage()` → `pool.usage` — it's a `@property` (export-source, clone-target).
2. `_stall_ticks` was a sample fiction (no such column) → added real
   `hydration_stall_ticks` Int field to the doctype.
3. `Server.provider` doesn't exist → use `Server.provider_type` in `preflight_checks`.
4. **`set_only_once` trap:** `VM.server` had `set_only_once: 1` in the JSON; Frappe's
   framework-level `validate_set_only_once()` runs BEFORE the controller `validate()`
   and has no bypass, so `flags.migrating` could never repoint `server`. Removed
   `set_only_once`; `server` stays immutable via the controller's
   `IMMUTABLE_AFTER_INSERT` (same pattern the RESIZE_MUTABLE fields use).
   **Spec follow-up:** spec/19 §1 should note a migration-repointed field must not
   carry `set_only_once`.

### Also learned (host facts)

- Both hosts have all migration deps (qemu-nbd, nbd-client, socat, nbd + dm_clone
  modules, kernel 6.8.0-88). Bootstrap dep additions confirmed present.
- Raw host-to-host TCP works both ways; a detached listener over SSH must use
  `systemd-run` (a bare `nohup … &` dies on session close).
- The runner's first invocation of a freshly-synced script may hit the
  staging→durable fallback (double Task: Failure then Success); subsequent runs use
  the durable path cleanly. Sync scripts before an e2e.
